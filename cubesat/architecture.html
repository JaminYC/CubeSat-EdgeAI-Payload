03Ciclo de vida de un scan sequence

Desde que el OBC envia START_CAPTURE hasta que el paquete de downlink queda listo para bajar. El ack al OBC es inmediato (< 10 ms); el trabajo pesado ocurre asincronico.

sequenceDiagram autonumber participant O as OBC participant I as i2c_slave participant Q as /run/cubesat/commands participant D as daemon participant C as FPMCapture participant P as Controller IA participant B as /var/cubesat O->>I: CMD_START_CAPTURE (0x02, mode=1) I->>Q: write cmd_UTC.json I-->>O: STATUS_OK (ack inmediato) D->>Q: inotify IN_CLOSE_WRITE D->>D: state = CAPTURING D->>C: capture_scan(mode=1) loop 25 angulos (5x5 OLED) C->>C: actualiza posicion LED C->>C: snap OV5647 C->>B: cap_NN.jpg.tmp → rename end C->>B: metadata.json atomico D->>D: state = PROCESSING D->>P: run(scan_id) P->>P: enhance (N2V) P->>P: segment (StarDist) P->>P: metricas + CSV P->>B: results/scan_UTC/ D->>D: state = EXPORTING D->>B: downlink/scan_UTC/{summary, thumb, telem} D->>D: state = IDLE Note over O,I: Mas tarde O->>I: CMD_GET_LAST_SUMMARY (0x04) I->>B: lee summary.json I-->>O: [OK, len, chunk0..N]

04Protocolo I2C con chunking sequence

Comandos chicos (≤ 16 B) se responden directo. Payloads grandes (thumbnail, CSV) se trocean en frames de 30 bytes con flag more.

sequenceDiagram autonumber participant O as OBC participant S as i2c_slave @ 0x42 Note over O,S: Comandos chicos → respuesta directa O->>S: write [0x01] CMD_GET_STATUS S-->>O: read 18 B [OK, 16, status_struct] Note over O,S: Payloads grandes → chunked (30 B/frame) O->>S: write [0x06] CMD_GET_THUMBNAIL S->>S: abre thumbnail.jpg (~100 KB) loop Mientras more=1 O->>S: read 32 B S-->>O: [OK, 30, more=1, chunk] O->>O: append end O->>S: read 32 B S-->>O: [OK, N, more=0, last_chunk] Note over O,S: Errores O->>S: write [0x04] (sin scan previo) S-->>O: [NO_DATA, 0] O->>S: write [0xFA] (opcode invalido) S-->>O: [UNKNOWN_CMD, 0]
Limites a 100 kHz: status = 2 ms, thumbnail 100 KB ~ 10 s, data.csv 1 MB ~ 100 s. Por eso el CSV es on-demand, no parte del downlink automatico.
